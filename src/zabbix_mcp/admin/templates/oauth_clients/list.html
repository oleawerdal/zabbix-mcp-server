OAuth Clients

Third-party MCP clients (ChatGPT custom apps, Claude Desktop, MCP Inspector, ...) that registered themselves against the embedded OAuth 2.1 authorization server.

{% if not oauth_enabled %}

OAuth is not enabled 🛈

Turn on the embedded OAuth 2.1 authorization server so MCP clients that auto-discover authentication (ChatGPT custom apps, Claude Desktop remote, MCP Inspector) can sign in against this deployment. Login uses the existing admin-portal users; no separate identity store.

{% if can_enable %}
Must be https:// with a valid DNS name (IP addresses won't work - public CAs don't issue certs for IPs). The setting is also writable in Settings → MCP Server → Public URL. No HTTPS yet? Run sudo ./deploy/install.sh request-tls --hostname mcp.example.com on the server to get a Let's Encrypt cert in one command.

The setting takes effect after the server restarts. The Restart needed badge appears in the page header once you save - click it to apply.

{% else %}

Only the admin role can change OAuth settings. Ask an admin to enable it, or edit config.toml on the server: set [oauth].enabled = true and [server].public_url to your externally-reachable HTTPS URL, then restart.

{% endif %}

Full walkthrough (TLS options, reverse-proxy snippets, role caps, refresh-token reuse detection) in docs/OAUTH.md.

{% else %}

Discovery 🛈

Clients auto-discover this server's OAuth endpoints via the well-known URLs. If a client cannot reach these, it cannot register or authenticate.
